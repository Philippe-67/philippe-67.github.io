Fiche pratique – Structure d’un projet backend .NET (dans une solution .sln)

Introduction

Cette fiche décrit la structure typique d’un projet backend .NET (API) contenu dans une solution Visual Studio (.sln). Tu y trouveras le rôle de chaque dossier et fichier important, pour mieux comprendre l’organisation d’un projet .NET moderne.

Structure générale d’un projet backend .NET

MonApi/                  ← Dossier du projet backend (API)
├── Controllers/         ← Contient les contrôleurs (logique des routes API)
├── Properties/          ← Fichiers de configuration du projet (ex : launchSettings.json)
├── appsettings.json     ← Fichier de configuration principal (paramètres, connexions...)
├── appsettings.Development.json ← Config spécifique au mode développement
├── MonApi.csproj        ← Fichier projet .NET (références, dépendances...)
├── Program.cs           ← Point d’entrée de l’application (configuration, démarrage)
├── bin/                 ← Dossier de sortie des fichiers compilés (généré automatiquement)
├── obj/                 ← Dossier temporaire pour la compilation (généré automatiquement)
└── ...                  ← Autres fichiers/dossiers selon les besoins

Détail des dossiers et fichiers

1. Controllers/

  • Contient les fichiers “contrôleurs” (ex : WeatherForecastController.cs).

  • Chaque contrôleur gère un ensemble de routes API (ex : /weatherforecast).

  • C’est ici que tu écris la logique pour répondre aux requêtes HTTP.

    Zoom sur le dossier Controllers/ et un contrôleur type

    Ce zoom t'explique comment fonctionne un contrôleur dans .NET : comment il reçoit une requête, traite la logique, et renvoie une réponse.

  • Un contrôleur est une classe C# qui hérite de ControllerBase ou Controller.

  • Chaque méthode publique (appelée "action") correspond à une route HTTP (GET, POST, etc.).

  • Exemple simplifié :

    [ApiController]
[Route("[controller]")]
public class WeatherForecastController : ControllerBase
{
    [HttpGet]
    public IEnumerable<WeatherForecast> Get()
    {
        // Logique pour retourner des données météo
    }
}

  • Les attributs [HttpGet], [HttpPost], etc. définissent le type de requête accepté.

  • Les contrôleurs sont le point d'entrée de toutes les requêtes API.

    2. Properties/

  • Contient généralement launchSettings.json.

  • Ce fichier définit comment l’application démarre en local (profils, ports, variables d’environnement).

    Zoom sur launchSettings.json

    Ce zoom t'explique comment personnaliser le lancement de ton application en local (port, variables d'environnement, profils de démarrage).

  • Fichier JSON généré automatiquement par Visual Studio ou VS Code.

  • Permet de définir plusieurs profils de lancement (ex : IIS Express, Kestrel, Docker).

  • Exemple de contenu :

    {
  "profiles": {
    "MonApi": {
      "commandName": "Project",
      "dotnetRunMessages": true,
      "launchBrowser": true,
      "applicationUrl": "https://localhost:5001;http://localhost:5000",
      "environmentVariables": {
        "ASPNETCORE_ENVIRONMENT": "Development"
      }
    }
  }
}

  • On peut y changer le port, ajouter des variables d'environnement, activer/désactiver l'ouverture automatique du navigateur, etc.

    3. appsettings.json

  • Fichier de configuration principal (format JSON).

  • Sert à stocker les paramètres de l’application (connexion à la base, clés API, options diverses).

    Zoom sur appsettings.json et appsettings.Development.json

    Ce zoom t'explique comment configurer ton application sans toucher au code, et comment séparer les réglages de développement et de production.

  • appsettings.json contient les paramètres globaux : chaînes de connexion, options personnalisées, clés API, etc.

  • appsettings.Development.json surcharge certains paramètres quand tu es en mode développement (ex : base de test, logs plus détaillés).

  • Exemple de contenu :

    {
  "ConnectionStrings": {
    "DefaultConnection": "mongodb://localhost:27017/monappdb"
  },
  "Logging": {
    "LogLevel": {
      "Default": "Information"
    }
  }
}

  • On accède à ces paramètres dans le code via l'injection de dépendances ou Configuration["MaCle"].

  • Ne jamais mettre de secrets de production dans ces fichiers : utiliser des variables d'environnement pour cela.

    4. appsettings.Development.json

  • Surchage les paramètres de appsettings.json en mode développement.

  • Pratique pour avoir des réglages différents en local (ex : base de données de test).

5. MonApi.csproj

  • Fichier projet .NET (XML).

  • Liste les dépendances NuGet, les frameworks utilisés, les options de build, etc.

  • C’est le “cœur” du projet pour .NET.

    Zoom sur MonApi.csproj

    Ce zoom t'explique comment ce fichier structure le projet, gère les dépendances et le framework utilisé.

  • C'est un fichier XML qui décrit :

    • Le nom du projet, la version du framework (ex : net8.0)
    • Les packages NuGet utilisés (ex : MongoDB.Driver, Swashbuckle.AspNetCore)
    • Les options de build, les fichiers inclus/exclus

  • Exemple de dépendance ajoutée :

    <ItemGroup>
  <PackageReference Include="MongoDB.Driver" Version="2.19.0" />
</ItemGroup>

  • Pour ajouter une librairie, on utilise la commande dotnet add package NomDuPackage qui modifie ce fichier.

  • Ce fichier ne doit jamais être modifié à la main sauf cas avancé.

    6. Program.cs

  • Point d’entrée de l’application.

  • Configure les services, les middlewares, et démarre le serveur web.

  • Depuis .NET 6, ce fichier est souvent très court grâce au “minimal hosting model”.

Zoom sur un Program.cs typique d’une API .NET moderne (exemple MaPremiereApplicationWeb)

Voici ce que l’on retrouve généralement :

  • Configuration du CORS : Permet d’autoriser les requêtes venant d’autres applications (ex : le frontend React). Exemple : csharp builder.Services.AddCors(options => { options.AddDefaultPolicy(policy => { policy.AllowAnyOrigin() .AllowAnyHeader() .AllowAnyMethod(); }); });

  • Ajout des services principaux :

    • AddControllers() : Active la gestion des routes via les contrôleurs (dossier Controllers/)
    • AddEndpointsApiExplorer() et AddSwaggerGen() : Préparent la documentation interactive de l’API (Swagger)

  • Construction de l’application :

    • var app = builder.Build(); crée l’application à partir de la configuration

  • Activation de Swagger en développement :

    • Permet de tester facilement l’API via une interface web

  • Activation du CORS :

    • app.UseCors(); autorise les requêtes du frontend

  • Middlewares standards :

    • UseHttpsRedirection() : Force l’utilisation du HTTPS
    • UseAuthorization() : Gère l’authentification/autorisation (si besoin)

  • Branchement des contrôleurs :

    • app.MapControllers(); relie toutes les routes définies dans le dossier Controllers/

  • Lancement de l’application :

    • app.Run(); démarre le serveur web

  • Définition de types internes (optionnel) :

    • Parfois, des types simples (ex : record WeatherForecast) sont définis à la fin du fichier pour des besoins de test ou d’exemple

Ce modèle rend la configuration claire, concise et adaptée à la plupart des projets d’API .NET modernes.

7. bin/

  • Dossier généré automatiquement à la compilation.
  • Contient les fichiers exécutables et les DLLs du projet.
  • À ne jamais versionner sur GitHub (à ignorer dans .gitignore).

8. obj/

  • Dossier temporaire utilisé lors de la compilation.
  • Contient des fichiers intermédiaires, jamais à modifier ni à versionner.

Autres éléments possibles

  • wwwroot/ : Dossier pour les fichiers statiques (images, JS, CSS) si besoin.
  • Migrations/ : Dossier créé par Entity Framework pour gérer les migrations de base de données.
  • Fichiers .http : Pour tester les routes API directement depuis VS Code.

Bonnes pratiques

  • Ne jamais modifier ou versionner les dossiers bin/ et obj/.
  • Documenter la configuration dans le README.md.
  • Organiser les contrôleurs et services dans des dossiers dédiés si le projet grandit.

Cette fiche t’aide à te repérer dans un projet backend .NET et à comprendre le rôle de chaque élément clé.
